This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

This is the `fffunKokkos` branch. It benchmarks Kokkos float-float (FF) arithmetic kernels, comparing device results against host `quadmath.h` references (~34 digits) to measure accuracy. FF arithmetic represents an FP64-class value as an unevaluated sum of two FP32s, giving ~14.4 decimal digits of precision (24-bit FP32 mantissa × 2 = 48 bits total). It produces two executables: one for 39 real math operations and one for 24 complex math operations.

**Hardware caveat:** the build is configured for sm_100 (Blackwell, ~2:1 FP32:FP64 throughput). Expect FF to be slower than native FP64 on this class of hardware — slowdown ≫ 1.0×. The FF win appears on consumer GPUs with 32:1 to 64:1 FP32:FP64 ratios.

**Precision deficit:** FF (~14.4 digits) approaches FP64 (~15.95 digits) from below — it does not match. The accuracy column caps at 14.0; algorithmically perfect FF results visibly trail FP64 by ~1.5 digits in the display.

## Build

The build requires Kokkos ≥5.1 with CUDA backend, GCC 13.3.0, CMake 3.28.3, and CUDA 12.9.1. On Argonne systems, load modules first:

```bash
source scripts/prepare.sh
```

Then build (clones and installs Kokkos 5.1.0, then builds the demo):

```bash
source scripts/build_with_kokkos.sh <install-dir>
```

This outputs `build/kokkos_ff_demo` and `build/kokkos_ff_demo_complex`, and generates `setup.sh` with the necessary environment variables (`KOKKOS_HOME`, `CMAKE_PREFIX_PATH`, `LD_LIBRARY_PATH`, etc.).

If Kokkos is already installed, build directly with CMake:

```bash
cmake -B build -DCMAKE_PREFIX_PATH=<kokkos-install-dir>
cmake --build build -j$(nproc)
```

## Running Benchmarks

```bash
# All real operations (single table output)
./build/kokkos_ff_demo --batch 500000 --repeats 5

# Single real operation
./build/kokkos_ff_demo --op sin --batch 1000000 --repeats 5

# All complex operations
./build/kokkos_ff_demo_complex --batch 500000 --repeats 5

# Single complex operation
./build/kokkos_ff_demo_complex --op exp --batch 1000000 --repeats 5
```

Arguments: `--op <name>`, `--batch N` (default: 1,000,000), `--repeats N` (default: 5), `--seed N` (default: 12345).

Scripts for batch runs: `scripts/run_all_ff_ops.sh` and `scripts/run_all_ff_complex_ops.sh`.

## Architecture

### Three-Layer Design

```
Demo Executables (src/demo_ff_real.cpp, src/demo_ff_complex.cpp)
    └─ Float-Float Math Library
       (third_party/include/ff_math.hpp, ff_complex.hpp)
       └─ Kokkos Runtime + CUDA Backend
```

### Float-Float Math Library (`third_party/include/`)

`ff_math.hpp` defines `ffloat { float hi; float lo; }` in `namespace quad::ffun`. The `hi` component holds the primary value; `lo` holds the rounding error from `hi`. Key algorithms:

- **TwoSum (Knuth)**: basis of `ffadd` / `ffsub` — error-free summation
- **TwoProduct (Dekker splitting)**: basis of `ffmul` / `ffdiv`. Splitter constant is `8193.0f` (= 2^13 + 1) for FP32, vs `134217729.0` (= 2^27 + 1) used in DD for FP64.
- **Constants**: constructed from IEEE 754 bit patterns via `make_ff()` for both host (`std::memcpy`) and device (`__int_as_float`). The constants block in `ff_math.hpp` is auto-generated by `scripts/gen_ff_constants.cpp` (Route A: each FF constant is the round-to-nearest-FF of Bailey's existing FP64 (hi, lo) DD pair).

All functions are `KOKKOS_INLINE_FUNCTION`. Algorithms ported from DDFUN (David H. Bailey, Lawrence Berkeley National Lab) by mechanical translation from the DD branch.

`ff_complex.hpp` defines `ffcomplex { ffloat re; ffloat im; }` with full complex math via `quad::ffun::` free functions.

### Demo Executables (`src/`)

Each demo follows the same pattern:
1. Parse args → generate operation-specific random inputs on host (FP64), then narrow to FP32 for the FF inputs
2. Compute `quadmath.h` host reference (ground truth, ~34 digits) from the FP64 inputs
3. Deep copy inputs to Kokkos device Views as `ffloat` and `double`
4. Run 2 warmup + N timed `Kokkos::parallel_for` launches with `Kokkos::fence()` after each
5. Deep copy results back, compute per-element relative error → convert to "digits of accuracy" (`-log10(rel_error)`, clamped to [0, 14])
6. Print formatted table: FF vs FP64 timing and accuracy statistics

Timing includes `Kokkos::fence()` overhead (synchronization cost), not raw kernel time.

Accuracy is reported as precise decimal digits: 14 = saturated at FF resolution (~14.4 digits max), 0 = completely wrong.

### Constants generator (`scripts/gen_ff_constants.cpp`)

A small (~30 LOC) host program. For each Bailey DD constant `(hi64, lo64)`:
1. `hi32 = (float)hi64`  — nearest FP32 of `hi64`
2. `residual = (hi64 - (double)hi32) + lo64`  — exact in FP64
3. `lo32 = (float)residual`  — nearest FP32 of the residual

The bit patterns of `(hi32, lo32)` are emitted as `make_ff(0xHHHHHHHHU, 0xLLLLLLLLU)` lines, which are pasted into `ff_math.hpp`. Run once and commit; do not regenerate during normal work.

### Validator (`scripts/test_ffmul.cpp`)

Standalone host program that exercises `ffmul` and `ffmulff` against FP64 ground truth. `ffmulff` should be bit-exact (zero error); `ffmul` should peak at ~1-2 ulp at FF resolution (max relative error ≲ 2e-14). Run after any change to `ffmul` or the Dekker splitter constant.

### Namespace

All FF types and functions live in `namespace quad::ffun` (parallel to `quad::ddfun::` on `ddfunKokkos` and `quad::cuda_fp128::` on `main`).

### Platform Constraint

`libquadmath` (host reference) is x86_64 only. CMake enforces this — the project will not build on ARM or other platforms.
